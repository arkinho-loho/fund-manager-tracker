# 基金经理持仓与观点跟踪

## 一、红线(继承 wb-finance-skill)

- **禁编造**: 持仓/观点数据缺失时标注"缺失",不补全
- **禁混淆**: 报告期 vs 披露日、加仓 vs 新建、净值 vs 累计净值
- **每个数据点带来源**(neodata) + 时间戳(报告期 YYYYQn)
- **不编造观点**: 观点原文未获取时标记"⚠️ 缺失",绝不杜撰

## 二、数据源路由

| 优先级 | 工具 | 用途 | 说明 |
|---|---|---|---|
| **第一** | `neodata-financial-search` | 经理/基金/持仓/排名全品类 | 主力数据源 |
| **第二** | `hithink-fundmanager-selector`(问财) | 经理交叉验证/条件筛选/补充业绩 | **同花顺问财**, 详见 references/iwencai-integration.md |
| **第三** | `announcement-search`(问财公告) | 基金/股票公告搜索(季报/年报/分红/回购等) | **同花顺公告搜索**, 升级观点获取链路,详见 references/iwencai-integration.md |
| **第四** | `天天基金网 API` | **历史季度持仓获取 + 季报公告原文** | **补位 neodata 缺失的历史持仓数据**,详见下方 |
| 禁用 | `westock-data` / `westock-tool` | 仅覆盖 ETF,不适用于主动管理型公募基金 | — |

### 第四数据源:天天基金网(历史持仓补位)

> **定位**: neodata 只能返回最新一期持仓数据(2026Q1),无法获取历史季度(2025Q4),导致跨期对比缺失。天天基金网提供完整的历史季度重仓股明细。

#### 历史持仓 API

```
URL: https://fundf10.eastmoney.com/FundArchivesDatas.aspx
参数: type=jjcc&code={基金代码}&topline=10&year={年份}&month={季度末月}
示例: type=jjcc&code=110011&topline=10&year=2025&month=12  → 2025Q4
      type=jjcc&code=110011&topline=10&year=2025&month=9   → 2025Q3
```

**调用方式**: `WebFetch(url)` → 解析 HTML table → 提取股票名称/代码/占净值比例

**覆盖范围**: A股所有公募基金(含QDII)

#### 季报公告页面

```
URL: https://fundf10.eastmoney.com/jjgg_{基金代码}_3.html
说明: 基金公告列表页，含季报/年报原文链接
用途: 观点原文提取的 L2 补充路径(详见 references/outlook-extraction.md)
```

### 三数据源协作策略

```
neodata(主力)          hithink-fundmanager-selector(问财)   announcement-search(问财公告)
    │                              │                               │
    ├── 经理基本信息 ◄── 交叉验证 ── 经理画像                       │
    ├── 在管基金列表 ◄── 补充校验 ── 产品清单                       │
    ├── 持仓明细(>2%)         (问财不覆盖)                          │
    ├── 观点原文(三级降级)   (原策略)                               │
    │   └── 升级为五级降级                                         │
    │       └── L1: neodata docData                                │
    │       └── L2: ←─ 公告搜索直达 ─────────── 基金季报/年报原文  │
    │       └── L3: WebSearch                                      │
    │       └── L4: WebFetch                                       │
    │       └── L5: 标记缺失                                       │
    ├── 各阶段收益率 + 同类排名 ◄── 补充 ── 收益率/排名/回撤        │
    ├── 规模/行业/排名等扩展维度          │                          │
    └──                                   ├── 条件筛选:找新经理     │
                                          └── 公告搜索 ────────── 重仓股动态
```

**核心原则**:
- **持仓只走 neodata**(问财/公告不提供持仓数据)
- **观点原文链路升级五级降级**: L1 neodata docData → L2 announcement-search(问财公告) → L3 WebSearch → L4 WebFetch → L5 标记缺失(详见 references/outlook-extraction.md)
- **经理信息以 neodata 为主,问财基金画像辅助验证**
- **业绩数据优先用 neodata,若缺失则用问财补充**
- **条件筛选(找新经理)走 hithink-fundmanager-selector**
- **公告搜索走 announcement-search**(基金季报原文直达、重仓股动态)
- 数据冲突时优先采信 neodata,标注分歧来源

**观点原文获取链路**(五级降级):
1. L1: neodata docData 试搜(默认 data_type=all 最大化召回) → 置信度:中
2. L2: **announcement-search 公告搜索**(问财直达,搜基金季报/年报) → 置信度:**高**
3. L3: WebSearch(不限站点,优先财经媒体) → 置信度:中
4. L4: WebFetch 抓取段落原文 → 置信度:**高**
5. L5: 标记缺失,继续执行持仓分析 → 置信度:无

### 基金定期报告来源(季报/中报/年报原文)

| 渠道 | URL | 权威性 | 在 Skill 中的角色 |
|---|---|---|---|
| **天天基金网** | `fundf10.eastmoney.com` | ⭐⭐⭐ 最完整的基金报告库 | **历史持仓 API(L4 补位) + L2/L4 观点原文** |
| **天天基金网持仓API** | `FundArchivesDatas.aspx` | ⭐⭐⭐ 结构化历史持仓 | **跨期对比主力**(neodata缺历史季度时使用) |
| **巨潮资讯网** | `www.cninfo.com.cn` | ⭐⭐⭐ 证监会指定信息披露平台 | L3 WebSearch 备选 |
| **证监会基金电子披露** | `fund.csrc.gov.cn` | ⭐⭐⭐ 官方披露平台 | 备选 |
| 各基金公司官网 | 如 `www.efunds.com.cn` | ⭐⭐ "信息披露"栏目 | 备选 |
| neodata docData | 内部 API | ⭐ 财经资讯/研报召回,非保证季报原文 | L1 试搜(最快但命中率低) |

> 💡 **说明**: 
> - **历史持仓**: neodata 只返回最新一期,天天基金网 `FundArchivesDatas.aspx` API 补充历史季度(2025Q4等),实现跨期持仓对比
> - **观点原文**: 天天基金网 + 巨潮资讯网两个站点基本能覆盖 95% 以上的公募基金定期报告原文
> - 业绩数据(收益率/同类排名)直接从 neodata 结构化接口获取,无需爬取报告原文

### 业绩数据来源

| 数据项 | neodata 接口 | 用途 |
|---|---|---|
| 各阶段收益率 | 基金业绩(近1月/3月/6月/1年/3年/成立来) | 业绩展示 |
| 同类排名 | 基金各阶段同类排名 | 排名百分位 |
| **业绩基准名称** | 基金基础信息(业绩比较基准字段) | **标注对标基准** |
| **业绩基准对比** | 基金基准收益率 | **超额收益** |
| **业绩基准排名** | 基金各阶段同类排名(基准同类别) | **基准排名百分位** |

## 三、触发与输入

### 触发词(口语识别)
- "跟踪基金经理 / 抄作业 / 经理持仓 / 经理观点"
- "X 最新持仓 / X 调仓 / X 重仓股"(X=经理名)
- "帮我看看 X 的季报 / 年报"

### 输入参数
| 参数 | 格式 | 默认值 | 说明 |
|---|---|---|---|
| 经理名 | 字符串(单个或批量) | 无(必填) | 如"张坤"或"名单全部" |
| 对比期数 | 数字 N | 1 | 对比最近 N 期(1=最新 vs 上一期) |
| 扩展维度 | 开关参数 | 全部关闭 | --include-scale / sector / ranking / turnover |
| 输出格式 | md \| html \| both | **both** | 默认生成 MD + HTML 双版本 |
| 指定基金 | 基金代码 | 自动筛选 | --fund 110011 (跳过标杆筛选) |
| 名单路径 | 文件路径 | watchlist.json | --list custom.json |

**默认名单**: `assets/manager-watchlist.json`(~30 位基金经理)

## 五、可选参数(扩展维度开关)

| 参数 | 默认 | 说明 | 数据源 |
|---|---|---|---|
| `--periods N` | 1 | 对比最近 N 期(1=最新 vs 上一期) | neodata 基金持仓明细 |
| `--format md\|html\|both` | **md** | 输出格式（默认仅 Markdown，HTML 不再维护） | — |
| `--include-scale` | off | 附规模变动(最近 N 期) | neodata 基金规模走势 |
| `--include-sector` | off | 附行业配置变化 | neodata 基金行业组合 |
| `--include-ranking` | off | 附同类排名详情 | neodata 基金各阶段同类排名 |
| `--include-turnover` | off | 附换手率(若可召回) | neodata(试探) |
| `--fund <code>` | 自动筛选 | 用户指定具体基金(跳过标杆筛选) | — |
| `--list <path>` | watchlist.json | 自定义名单文件路径 | assets/ |

> 📌 **业绩展示为默认功能**(非可选参数):每次报告都会包含近1月/3月/6月/1年/3年/成立来收益率 + 同类排名。

**口语映射示例**:
- "对比最近2期" → `--periods 2`
- "带规模和行业配置" → `--include-scale --include-sector`
- "用易方达蓝筹精选这只" → `--fund 005827`
- "只要Markdown" → `--format md`

## 六、错误处理与降级

| 故障场景 | 处理方式 | 是否阻断 |
|---|---|---|
| neodata 鉴权失败 | 按 SKILL.md 重试 1 次(connect_cloud_service);仍失败则切换问财 | ✅ 阻断 neodata 但不阻断整体 |
| 经理未找到(neodata) | 利用问财辅助查询;仍空则标注"未找到" | ❌ 不阻断(跳过该经理) |
| 经理未找到(neodata+问财均无) | 标注"未找到(两数据源均无数据)" | ❌ 不阻断(跳过该经理) |
| 问财 API 鉴权失败 | 提示配置 IWENCAI_API_KEY,降级为仅 neodata | ❌ 不阻断(neodata 照常) |
| 经理已离任 | is_active=false,标注"已离任",终止该经理流程 | ❌ 不阻断(正常终止) |
| 某期持仓数据缺失 | 对比时跳过该期,标注"{period} 数据缺失" | ❌ 不阻断(其他期照常) |
| 观点 L1-L3 全失败 | 标"⚠️ 观点原文未获取",继续执行持仓分析 | ❌ **绝对不阻断** |
| 基金元数据不全 | 用已有字段降级排序,标注警告 | ⚠️ 可能影响标杆选择准确性 |
| WebSearch/WebFetch 失败 | 观点降级到 L4;不重试避免阻塞 | ❌ 不阻断 |
| 批量场景单经理异常 | 记录"该经理处理失败:{原因}",继续下一位 | ❌ 不阻断 |
| 两数据源数据冲突 | 优先采信 neodata,标注分歧 | ❌ 不阻断(如实标注) |

**核心原则**:
- **部分缺失不阻塞整体**:持仓分析是强能力,观点是弱能力,两者解耦
- **显式标注而非静默跳过**:每个缺失点都明确告知用户
- **置信度分层**:高(一手原文)/中(docData或媒体)/无(缺失),让用户判断可信度

### 已离职经理的生命周期管理

当 `validate_manager` 判定经理已离职时(在管基金列表为空):

```
经理已离职(在管列表为空)
        │
        ▼
┌──────────────────────────────────────────────────────────┐
│ 保留在观察名单(assets/manager-watchlist.json),不删除     │
│ 标记 is_active=false,记录最后活跃日期                    │
└──────────────────────────────────────────────────────────┘
        │ (下次启动技能时)
        ▼
┌──────────────────────────────────────────────────────────┐
│ Step A: 再次 validate_manager 查是否入职新公募基金       │
│  neodata: "基金经理 {name} 最新从业履历"                 │
│  → 如果已入职新公募:                                      │
│     1. 更新 watchlist.json 中的公司信息和 style_tags      │
│     2. 重新 select_benchmark_fund(新在管基金)            │
│     3. 执行完整六步流程                                   │
│     4. 在报告中标注"已在{新公司}复职"                    │
│  → 如果仍未入职公募: 进入 Step B                         │
└──────────────────────────────────────────────────────────┘
        │ (仍未入职公募)
        ▼
┌──────────────────────────────────────────────────────────┐
│ Step B: 查询是否入职私募基金                             │
│  WebSearch: "{name} 基金经理 入职 私募"                  │
│  → 如果确认已入职私募(有明确新闻/公告):                   │
│     1. 在 watchlist.json 中标记 need_remove=true          │
│     2. 从活动跟踪名单中移除                               │
│     3. 在报告中标注"已入职{私募名},已从跟踪名单中移除"   │
│  → 如果无明确去向:                                        │

```

### 全局一致性更新规则

> **核心原则**: 用户要求修改报告中任一数据时，必须扫描该项目的**全部输出文件**(MD、HTML、JSON)，查找所有关联内容并一并更新。不要只改用户指出的那一处。

#### 变更传播映射表

| 用户说 | 必须改的文件 |
|--------|-------------|
| "更新X经理的业绩/持仓" | watchlist.json + MD(表格行+统计数) + HTML(表格行+JS data+summary-grid) |
| "X经理名字/公司写错了" | watchlist.json + MD(全文搜索替换) + HTML(全文搜索替换) + JSON数据文件(全文) |
| "X经理不算异常/已修正" | MD(错误表+汇总) + HTML(错误表+JS status字段+summary-grid数字+章节标题) |
| "X和Y是同一个人" | 删除旧条目 + 合并style_tags + 更新新条目 + MD/HTML全文查旧名 |
| "新增/删除经理" | watchlist.json + MD(总表行+统计数) + HTML(总表行+JS数据+summary-grid) |
| "is_active变更" | watchlist.json + MD/HTML中该经理行内状态图标 |

#### 编辑后必执行检查清单

1. **关键词搜旧值** — grep旧值 MD/HTML/JSON → 确认只在"已修正"说明中出现
2. **汇总数字校核** — MD执行概要数字 vs 明细表行数 → 必须一致
3. **MD/HTML双版本对齐** — 两款报告中的同一数据项必须相同
4. **watchlist.json做主** — 报告中的 is_active/company/benchmark_fund 以 watchlist.json 为准
5. **类型规则** — is_active 必须是 true/false，不允许字符串

#### 常见错误（禁止）

- 用户说"更新王冠桥的MD" → 不得只改MD，须同步 MD+HTML+JSON+watchlist
- 用户说"把3改成1" → 不得只改指的那一个"3"，须全文所有相关数字一并更新
- 只改HTML表格数据但忘了改JS data数组 → 两处必须同步

## 七、示例场景

### 场景 1:查询单个经理的最新情况
**用户**: "帮我看看张坤最新的持仓和观点"
**执行**:
1. validate_manager("张坤") → 在职,在管 N 只
2. select_benchmark_fund(...) → 选出标杆(如易方达蓝筹精选)
3. extract_holdings([标杆], periods=1) → 最新期+上期重仓(>2%)
4. extract_outlook(标杆, 最新期) + extract_performance(标杆) → 观点(降级)+业绩
5. compare_periods(...) → 持仓变动 + 观点变化
6. format_report → 单经理卡片报告(MD + HTML 双版本)

**输出**: 含档案/标杆/业绩表现/持仓表/变动表(✚↑↓✖=符号)/观点diff 的 MD+HTML 双版本报告

### 场景 2:批量查询名单中所有经理
**用户**: "把我名单里的经理都过一遍"
**执行**:
1. 解析 manager-watchlist.json(~30 位)
2. **串行**对每位执行场景 1 全流程(避免并发压垮 neodata)
3. 单位失败不阻塞,记录失败清单
4. 汇总为批量报告:
   - 顶部汇总表(每经理一行:在职/标杆/近1年收益率+排名/前三大重仓/变动数/态度)
   - 逐位详情(可折叠)

**输出**: 汇总表 + 逐位详情的 MD+HTML 双版本;末尾附"⚠️ 处理失败:{名单}"

### 场景 3:指定对比最近 2 期变化
**用户**: "对比杨锐文最近2期的持仓和观点变化"
**执行**:
1. validate_manager("杨锐文")
2. select_benchmark_fund(...)
3. extract_holdings([标杆], periods=2) → 取最近 3 期(最新+2个对比期)
4. extract_outlook × 3期 + extract_performance(标杆)
5. compare_periods(...) → 两段对比(期2→期1,期3→期2),输出变动趋势
6. format_report → 强调"变化趋势"维度

**输出**: 含业绩表现+两段变动对比表 + 观点态度演进(如 乐观→中性→谨慎)的 MD+HTML 双版本

### 场景 4:条件搜索新经理并跟踪(问财特色功能)
**用户**: "帮我找管理规模超100亿、近1年收益排名前20%、近3年最大回撤小于-20%的基金经理"
**执行**:
1. **问财筛选**: 调用 hithink-fundmanager-selector 执行条件搜索
   ```python
   python3 scripts/cli.py --query "管理规模超100亿、近1年收益排名前20%、近3年最大回撤小于-20%的基金经理" --limit 20
   ```
2. 解析返回结果,提取符合条件的经理名单
3. **自动加入临时跟踪列表**,对每位经理执行场景 1 完整流程
4. 批量输出跟踪报告,末尾附"问财发现"标签

**输出**: 问财条件筛选结果 + 自动跟踪报告;提示用户是否将新经理加入 watchlist.json

### 场景 5:交叉验证经理信息
**用户**: "帮我确认一下张坤的最新信息"
**执行**:
1. neodata: 查询张坤的履历和在管基金
2. **问财**: 同步查询张坤的经理画像(管理规模/收益率/排名/回撤)

   - 管理总规模 → 差异大小
   - 各阶段收益率 → 是否吻合

**输出**: 标注"✅ 数据一致"或"⚠️ 存在差异(neodata:XX 亿 vs 问财:XX 亿)"

## 十、使用建议与最佳实践

1. **首次使用前**: 确保 `assets/manager-watchlist.json` 已填好你的跟踪名单
2. **最佳查询时机**: 季报披露后 1-2 周(T+15 左右),此时数据最完整
3. **批量查询注意**: ~30 位经理串行预计 10-20 分钟,耐心等待
4. **解读变动信号**:
   - ✚ 新建仓: 经理新看好的方向(重点研究)
   - ✖ 清仓: 经理不再看好(警惕风险)
   - ↑ 大幅加仓: 强烈看多(可能已有较大涨幅)
   - ↓ 大幅减仓: 开始撤退(关注是否已兑现)
   - = 持平: 长期持有确认(核心仓位稳定)
5. **观点置信度判断**:
   - 高(WebFetch一手原文):可直接采信
   - 中(neodata/docData/WebSearch):需交叉验证
   - 无(缺失):仅参考持仓变动,不参考观点
6. **问财集成要点**：
   - 经理信息/业绩：neodata(主) + 问财(辅)，问财可作为 neodata 数据缺失时的 fallback
   - 持仓和观点：仅 neodata，问财不覆盖此维度
   - 条件筛选找新经理：用问财发现并自动加入跟踪列表
   - 数据冲突时优先采信 neodata，如实标注分歧来源
   - 调用方法：`python3 {skill_path}/scripts/cli.py --query "..."`
